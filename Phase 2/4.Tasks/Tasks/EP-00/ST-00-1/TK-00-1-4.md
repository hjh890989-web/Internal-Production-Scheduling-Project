```yaml
# infrastructure/docker-compose.yml
# Docker Compose v2 (Compose Specification 준수, version 키 생략 가능)
name: scheduling

services:
  # ============================================================
  # 1. PostgreSQL 16 (TK-00-1-1)
  # ============================================================
  postgres:
    image: postgres:16-alpine
    container_name: scheduling-postgres
    # ... TK-00-1-1 정의 그대로 (생략)

  # ============================================================
  # 2. Redis 7 (TK-00-1-2)
  # ============================================================
  redis:
    image: redis:7-alpine
    container_name: scheduling-redis
    # ... TK-00-1-2 정의 그대로

  # ============================================================
  # 3. Backend (placeholder — ST-00-2에서 실제 Spring Boot로 교체)
  # ============================================================
  backend:
    image: nginx:1.25-alpine    # ST-00-2 완료 전 placeholder
    container_name: scheduling-backend
    expose:
      - "8080"
    healthcheck:
      test: ["CMD", "wget", "--spider", "-q", "http://localhost:80/"]   # placeholder healthcheck
      interval: 10s
      timeout: 3s
      retries: 5
      start_period: 10s
    depends_on:
      postgres:
        condition: service_healthy
      redis:
        condition: service_healthy
    restart: unless-stopped
    networks:
      - scheduling-net

  # ============================================================
  # 4. Frontend (placeholder — ST-00-3에서 실제 React 빌드로 교체)
  # ============================================================
  frontend:
    image: nginx:1.25-alpine
    container_name: scheduling-frontend
    volumes:
      - frontend-static:/usr/share/nginx/html
    healthcheck:
      test: ["CMD", "wget", "--spider", "-q", "http://localhost:80/"]
      interval: 10s
      timeout: 3s
      retries: 3
      start_period: 5s
    restart: unless-stopped
    networks:
      - scheduling-net

  # ============================================================
  # 5. NGINX (TK-00-1-3) — depends_on 활성
  # ============================================================
  nginx:
    image: nginx:1.25-alpine
    container_name: scheduling-nginx
    # ... TK-00-1-3 정의 그대로 + depends_on:
    depends_on:
      backend:
        condition: service_healthy
      frontend:
        condition: service_healthy

volumes:
  postgres-data:
    driver: local
  redis-data:
    driver: local
  frontend-static:
    driver: local

networks:
  scheduling-net:
    driver: bridge
    name: scheduling-net
```

> **참고**: TK-00-1-1·1-2·1-3의 개별 서비스 정의는 본 통합 파일에서 합치되, 중복 정의 회피. `depends_on` + `condition: service_healthy`로 부팅 순서 강제 (Compose v2 기능).

#### B. `Makefile` (운영 편의)

```makefile
# infrastructure/Makefile
# 사용: cd infrastructure && make up

.PHONY: up down logs ps restart clean status reset cert

up:
	docker compose --env-file .env up -d

down:
	docker compose down

logs:
	docker compose logs -f --tail=100

ps:
	docker compose ps

restart:
	docker compose restart

clean:
	docker compose down -v   # ⚠️ volume까지 삭제 — 데이터 손실

status:
	@echo "=== Container Status ==="
	@docker compose ps
	@echo ""
	@echo "=== Healthcheck Detail ==="
	@for c in scheduling-postgres scheduling-redis scheduling-nginx scheduling-backend scheduling-frontend; do \
		echo "$$c: $$(docker inspect --format='{{.State.Health.Status}}' $$c 2>/dev/null || echo 'not running')"; \
	done

reset: clean cert up

cert:
	bash nginx/scripts/generate-dev-cert.sh
```

#### D. 통합 검증 스크립트 (선택 — `tools/infra_validation/validate_compose.sh`)

```bash
#!/usr/bin/env bash
# 5개 컨테이너 부팅 + 모든 healthcheck 통과 자동 검증
set -euo pipefail

cd "$(dirname "$0")/../../infrastructure"

echo "🧹 기존 컨테이너 정리..."
docker compose down 2>/dev/null || true

echo "🚀 5개 컨테이너 부팅..."
docker compose --env-file .env up -d

echo "⏳ healthcheck 대기 (최대 60초)..."
timeout 60 bash -c '
    until [[ $(docker compose ps --format json | jq -r ".[].Health" | sort -u | wc -l) -eq 1 ]] \
       && [[ $(docker compose ps --format json | jq -r ".[].Health" | head -1) == "healthy" ]]; do
        sleep 2
    done
' || {
    echo "❌ healthcheck 타임아웃 — 60초 내 모든 컨테이너 healthy 실패"
    docker compose ps
    exit 1
}

echo "✅ 5개 컨테이너 모두 healthy"
docker compose ps

echo ""
echo "🔍 통합 통신 검증..."
curl -kfs https://localhost/health > /dev/null || { echo "❌ NGINX health 실패"; exit 1; }
echo "  ✅ NGINX → /health"

# PG 접속 (app_user)
docker compose exec -T postgres psql -U app_user -d scheduling -c "SELECT 1" > /dev/null \
    && echo "  ✅ PostgreSQL → SELECT 1" \
    || { echo "❌ PG 실패"; exit 1; }

# Redis PING
docker compose exec -T redis redis-cli -a "$REDIS_PASSWORD" PING | grep -q PONG \
    && echo "  ✅ Redis → PING/PONG" \
    || { echo "❌ Redis 실패"; exit 1; }

echo ""
echo "🎉 모든 검증 통과 — Sprint 0 DoD 항목 1 충족"
```

## :memo: Implementation Notes
- **기술 힌트**:
  - Compose v2의 `name: scheduling` (top-level) → 프로젝트 이름 명시. `docker compose -p` 없이도 컨테이너 prefix 통일
  - `condition: service_healthy`는 v2.3+ 기능 — `version: "3.8"` 키 없이도 동작 (Compose Specification)
  - `--env-file .env`는 명시적 (자동 로드도 되지만 명시가 안전)
- **유사 패턴**:
  - STG·PROD compose는 본 파일 + `docker-compose.staging.yml` overlay (`docker compose -f docker-compose.yml -f docker-compose.staging.yml up`)
  - CI/CD (EP-32 ST-32-1)에서 본 통합 검증 스크립트 호출
- **표준 참조**:
  - Docker Compose Specification: https://compose-spec.io/
  - Compose v2 Best Practices: https://docs.docker.com/compose/compose-file/
