### 핵심 로직

#### A. `FileStabilityChecker.java` — 핵심

```java
package com.scheduling.order.watcher;

import lombok.extern.slf4j.Slf4j;
import org.springframework.stereotype.Component;

import java.io.IOException;
import java.io.RandomAccessFile;
import java.nio.channels.FileChannel;
import java.nio.channels.FileLock;
import java.nio.channels.OverlappingFileLockException;
import java.nio.file.*;
import java.time.Duration;

/**
 * 파일이 안정 상태(쓰기 완료)에 도달했는지 검증.
 * SMB·NFS 환경에서 "복사 중 ingest" 버그 차단.
 */
@Slf4j
@Component
public class FileStabilityChecker {

    private static final int MAX_SAMPLES = 3;
    private static final Duration SAMPLE_INTERVAL = Duration.ofSeconds(1);

    /**
     * 파일이 stable 상태인지 검증.
     * - 크기 2회 연속 동일
     * - 파일 락 시도 성공 (다른 프로세스가 점유 중이 아님)
     *
     * @return true if stable (안전하게 ingest 가능)
     */
    public boolean isStable(Path file) {
        if (!Files.exists(file) || !Files.isRegularFile(file)) {
            return false;
        }

        // 1. 크기 안정화 sampling
        if (!sizeStabilized(file)) {
            log.debug("File size still changing: {}", file);
            return false;
        }

        // 2. 파일 락 시도 (Windows·SMB에서 효과적, Linux는 advisory)
        if (!canAcquireLock(file)) {
            log.debug("File locked by another process: {}", file);
            return false;
        }

        return true;
    }

    private boolean sizeStabilized(Path file) {
        long previousSize = -1;
        int stableCount = 0;

        for (int i = 0; i < MAX_SAMPLES; i++) {
            try {
                long currentSize = Files.size(file);
                if (currentSize == previousSize) {
                    stableCount++;
                    if (stableCount >= 2) {
                        return true;             // 2회 연속 동일 → stable
                    }
                } else {
                    stableCount = 0;
                }
                previousSize = currentSize;

                Thread.sleep(SAMPLE_INTERVAL.toMillis());
            } catch (NoSuchFileException e) {
                // 파일이 sampling 중 삭제됨 (취소된 복사 등)
                return false;
            } catch (IOException | InterruptedException e) {
                if (e instanceof InterruptedException) {
                    Thread.currentThread().interrupt();
                }
                return false;
            }
        }
        return false;   // MAX_SAMPLES 안에 안정화 못 됨 → 폴링에서 재시도
    }

    private boolean canAcquireLock(Path file) {
        // Linux는 advisory lock만 — 다른 프로세스가 락 안 잡으면 항상 성공
        // Windows·SMB는 mandatory lock — 정확히 검출
        try (RandomAccessFile raf = new RandomAccessFile(file.toFile(), "r");
             FileChannel channel = raf.getChannel()) {

            try (FileLock lock = channel.tryLock(0, Long.MAX_VALUE, true)) {  // shared lock
                return lock != null;
            } catch (OverlappingFileLockException e) {
                return false;
            }
        } catch (IOException e) {
            log.debug("Could not acquire lock on {}: {}", file, e.getMessage());
            return false;
        }
    }
}
```

#### B. `StableFileFilter.java` — Spring Event 필터

```java
package com.scheduling.order.watcher;

import lombok.RequiredArgsConstructor;
import lombok.extern.slf4j.Slf4j;
import org.springframework.context.ApplicationEventPublisher;
import org.springframework.context.event.EventListener;
import org.springframework.core.annotation.Order;
import org.springframework.stereotype.Component;

import java.time.Duration;

/**
 * FileDiscoveredEvent 발생 시 stability 검증 후 다음 핸들러로 위임.
 * 본 필터는 @EventListener 우선순위 HIGH(1) — TK-01-3-2의 FileIngestQueueService(@Order 10) 전에 실행
 */
@Slf4j
@Component
@RequiredArgsConstructor
public class StableFileFilter {

    private final FileStabilityChecker checker;
    private final ApplicationEventPublisher publisher;

    @EventListener
    @Order(1)
    public void onRawDiscovered(RawFileDiscoveredEvent event) {
        // TK-01-3-1이 RawFileDiscoveredEvent를 발행하도록 변경 (구분 위해)
        // 또는 본 필터 내에서 직접 stability 체크 후 FileDiscoveredEvent 재발행

        var path = event.filePath();
        if (!checker.isStable(path)) {
            log.info("File not yet stable, skip: {} — will retry via @Scheduled poller", path);
            return;
        }

        // stability 통과 → 정식 FileDiscoveredEvent 발행
        publisher.publishEvent(new FileDiscoveredEvent(
            path, event.sizeBytes(), event.discoveredAt(), event.source()
        ));
    }
}
```

> **설계 변경**: TK-01-3-1의 `FolderWatcherService`·`ScheduledFolderPoller`는 이제 `RawFileDiscoveredEvent`(stability 미검증)를 발행하고, `StableFileFilter`가 검증 후 정식 `FileDiscoveredEvent`를 재발행. TK-01-3-2는 변경 없음 (정식 이벤트만 수신).

### 측정 가능 기준

- [ ] **크기 변화 중 파일 skip**: 1초 간격 3회 sampling 안에 안정화 안 되면 RawEvent 폐기 (정식 이벤트 0)
- [ ] **2회 연속 동일 크기 → stable 판정**: 크기 변화 멈춘 직후 ≤ 3초 내 안정화 인식
- [ ] **파일 락 점유 시 skip**: 다른 프로세스가 exclusive lock 잡고 있으면 skip
- [ ] **다음 폴링에서 재시도**: skip된 파일은 다음 60s 폴링에서 다시 평가 → stable 시 진입
- [ ] **파일 sampling 중 삭제**: 안정성 검증 중 파일 삭제 시 false 반환, 예외 미발생
- [ ] **빈 파일(0 bytes)**: stable 판정 → 정식 이벤트 발행. 후속 파싱(TK-01-1-1)에서 ExcelParseException으로 거부
- [ ] **대용량 파일(100MB)**: 안정화 후 lock 검사 ≤ 2초 (메모리 mapping 없이 단순 open·close)
- [ ] **`StableFileFilter` 통과율 측정**: Prometheus 메트릭으로 raw 발행 / stable 통과 / skip 카운트 노출
- [ ] **장애 격리**: stability checker 예외 시에도 watcher 전체 동작 중단 X

### 회귀 시나리오 (T-I)

- **케이스 1 (쓰는 중)**: 빈 파일 생성 → 5초 동안 1MB씩 append → 정식 이벤트 0건 (RawEvent N건이지만 stable 미통과)
- **케이스 2 (쓰기 완료)**: 1MB 파일 한 번에 생성 → 즉시 stable 판정 (sampling 2~3초 후)
- **케이스 3 (락 점유)**: `FileChannel.lock()` (exclusive) 다른 thread에서 보유 → skip → 락 해제 후 다음 polling에서 통과
- **케이스 4 (sampling 중 삭제)**: 1초 후 파일 삭제 → false 반환, 예외 없음
- **케이스 5 (50MB 대용량)**: 10초간 streaming write → 완료 후 정상 stable
- **케이스 6 (빈 파일)**: 0-byte .xlsx → stable, 후속 단계에서 ExcelParseException 처리
- **케이스 7 (다중 파일 동시)**: 10개 파일 동시 drop → 각각 독립적으로 stability 평가
- **케이스 8 (NFS 환경)**: NFS-mounted 폴더에서 동일 시나리오 → 동일 동작 (락 동작 차이로 일부 false-positive 허용)

## :memo: Implementation Notes

- **기술 힌트**:
  - `FileChannel.tryLock(0, Long.MAX_VALUE, true)` — shared lock (다른 reader는 허용, writer만 차단)
  - `Files.size()`는 NFS에서 latency 가능 — sampling 간격이 효과적
  - **대안 컨벤션**: 영업 부서가 `report.xlsx` 복사 완료 후 `report.xlsx.ok` 빈 파일 생성 → watcher는 `.ok` 발견 시에만 처리 (Phase 1.1+)
  - **lock 검출 한계**: macOS·Linux의 advisory lock은 cooperating processes 사이에서만 동작
- **유사 패턴**:
  - MES 실적 자동 수신 watcher (Phase 2+) 도 동일 stability 패턴
- **표준 참조**:
  - POSIX file locking: https://pubs.opengroup.org/onlinepubs/9699919799/functions/fcntl.html
  - Java FileChannel.tryLock: https://docs.oracle.com/en/java/javase/21/docs/api/java.base/java/nio/channels/FileChannel.html
- **참고 외부 자료**:
  - "Detecting file write completion" — Stack Overflow patterns
