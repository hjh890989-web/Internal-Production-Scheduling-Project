#### B. Quality Gate 정의 (SonarQube UI 또는 API)

```bash
# 첫 부팅 후 한 번 실행 (또는 SonarQube UI에서 수동)
SONAR_URL="http://localhost:9000"
SONAR_TOKEN="<admin token>"

# Quality Gate 생성
curl -u "$SONAR_TOKEN:" -X POST "$SONAR_URL/api/qualitygates/create?name=Scheduling"

# 조건 추가
# 1. 신규 코드 커버리지 ≥80%
curl -u "$SONAR_TOKEN:" -X POST "$SONAR_URL/api/qualitygates/create_condition" \
    --data-urlencode "gateName=Scheduling" \
    --data-urlencode "metric=new_coverage" \
    --data-urlencode "op=LT" \
    --data-urlencode "error=80"

# 2. 신규 중복 ≤3%
curl -u "$SONAR_TOKEN:" -X POST "$SONAR_URL/api/qualitygates/create_condition" \
    --data-urlencode "gateName=Scheduling" \
    --data-urlencode "metric=new_duplicated_lines_density" \
    --data-urlencode "op=GT" \
    --data-urlencode "error=3"

# 3. 신규 Critical 코드 스멜 = 0
curl -u "$SONAR_TOKEN:" -X POST "$SONAR_URL/api/qualitygates/create_condition" \
    --data-urlencode "gateName=Scheduling" \
    --data-urlencode "metric=new_critical_violations" \
    --data-urlencode "op=GT" \
    --data-urlencode "error=0"

# 4. 신규 보안 핫스팟 review = 100%
curl -u "$SONAR_TOKEN:" -X POST "$SONAR_URL/api/qualitygates/create_condition" \
    --data-urlencode "gateName=Scheduling" \
    --data-urlencode "metric=new_security_hotspots_reviewed" \
    --data-urlencode "op=LT" \
    --data-urlencode "error=100"

# 기본 Quality Gate로 설정
curl -u "$SONAR_TOKEN:" -X POST "$SONAR_URL/api/qualitygates/set_as_default" \
    --data-urlencode "name=Scheduling"
```

> CASC plugin (sonar-config-as-code)으로 위 설정도 git 추적 가능. Phase 1.0 후 적용 권장.

#### E. Jenkinsfile 단계 추가

```groovy
// Jenkinsfile.backend (Test 단계 뒤)
stage('SonarQube Analysis') {
    when { not { branch 'feature/throwaway/*' } }
    steps {
        dir('backend') {
            withSonarQubeEnv('SonarQube') {
                sh """
                    ./gradlew sonar \
                        -Dsonar.projectVersion=${env.APP_VERSION} \
                        -Dsonar.branch.name=${env.BRANCH_NAME} \
                        --no-daemon
                """
            }
        }
    }
}

stage('Quality Gate') {
    steps {
        timeout(time: 5, unit: 'MINUTES') {
            waitForQualityGate abortPipeline: true   // 실패 시 빌드 중단
        }
    }
}
```

```groovy
// Jenkinsfile.frontend (Test 단계 뒤)
```

## :test_tube: Acceptance Criteria

**1차 검증 방법**: T-I (Integration) + A (Analysis)

### 측정 가능 기준
- [ ] SonarQube 컨테이너 부팅 + healthcheck 통과 (90초 이내)
- [ ] http://localhost:9000 진입 → admin 로그인 → Quality Gate "Scheduling" 활성 확인
- [ ] Backend `./gradlew sonar` 실행 → 분석 완료 + Quality Gate 결과 표시
- [ ] **커버리지 80% 미달 시 빌드 FAILED** (의도적 테스트 삭제로 검증)
- [ ] **Critical 코드 스멜 1개 추가 시 빌드 FAILED** (의도적 빈 catch 블록 등)
- [ ] **신규 중복 3% 초과 시 빌드 FAILED**
- [ ] Frontend `sonar-scanner` 동일 동작
- [ ] `waitForQualityGate` 결과 = OK 일 때만 다음 단계 진행
- [ ] PR별 SonarQube 분석 결과가 PR 페이지에 표시 (`branch.name` 활용)

### 회귀 시나리오 (T-I)
- **케이스 1 (정상)**: 80% 커버리지·중복 ≤3%·Critical 0 → Quality Gate OK → SUCCESS
- **케이스 2 (커버리지)**: 의도적 테스트 삭제로 75% 커버리지 → Quality Gate FAILED → 빌드 중단
- **케이스 3 (Critical)**: 의도적 빈 catch 추가 → FAILED
- **케이스 4 (브랜치 분석)**: feature 브랜치 분석 결과가 main 결과를 덮어쓰지 않음
