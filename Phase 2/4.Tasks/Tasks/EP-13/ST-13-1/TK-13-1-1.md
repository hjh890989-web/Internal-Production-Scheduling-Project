### Flyway `V4_5__add_vc_schedule_intraday_unique.sql`

```sql
-- 사전 점검 — TK-13-1-2의 사전 점검 SQL 통과 후에만 본 마이그레이션 적용
-- (운영 가이드: 사전 점검 FAIL 시 마이그레이션 중단 + 위반 row 정리)

-- BR-V07: 같은 (machine_id, slot_position, production_date) 슬롯에 다른 hose 배치 불가
-- DEFERRABLE INITIALLY DEFERRED — Allocator의 row swap 패턴 허용
ALTER TABLE vc_schedule
    ADD CONSTRAINT uk_vc_schedule_intraday_lock
    UNIQUE (machine_id, slot_position, production_date)
    DEFERRABLE INITIALLY DEFERRED;

COMMENT ON CONSTRAINT uk_vc_schedule_intraday_lock ON vc_schedule IS
    'EP-13 BR-V07 당일 락: 동일 (machine, slot, date)에 단일 hose만 (DEFERRABLE — swap 허용)';

-- 인덱스는 UNIQUE 제약이 자동 생성 — 별도 명시 불필요

-- 운영 검증 SQL:
--   SELECT machine_id, slot_position, production_date, COUNT(*)

--   GROUP BY 1,2,3
--   HAVING COUNT(*) > 1;
-- 본 쿼리 결과 0건이면 정합 (TK-13-1-2가 자동화)
```

## :shield: Risk Mitigation

- DEFERRABLE 없으면 row swap (delete+insert in same tx) 시 false positive — 본 옵션이 필수
- 기존 데이터 violation → 마이그레이션 실패 — TK-13-1-2 사전 점검이 차단
- 운영 환경 마이그레이션은 staging 우선 검증 — DBA 승인

## :memo: Implementation Notes

- `(machine_id, slot_position, production_date)` 3 필드 unique — `hose_id`는 제외 (같은 슬롯에 다른 hose가 들어가면 위반)
- WBS는 hose_id 포함 4 필드로 명시 (`machine_id, slot_position, production_date, hose_id`) — 그러나 hose_id 포함 시 same hose 다중 row만 차단 (다른 hose는 허용) — BR-V07 의도와 반대
- 본 구현은 3 필드로 — 운영팀 확정 후 final (PR 리뷰에서 합의)
